# 18. Check In Success Points

## Scope

Successful non-QR check-in, congratulations modal, points display, and post-success state.

## Preconditions

- Logged-in user.
- Location permission granted.
- Device location inside campaign check-in range.
- Store is open.
- Campaign has point multiplier fields configured.

## Scenario 18.1: Successful check-in

- Open a ready check-in item.
- Tap check-in.
- Check-in API should be called.
- Congratulations modal should open on success.
- Modal should show congratulations title.
- Earned points should be displayed.
- Brand/store logo should display when available.
- Store name/address should display when available.
- Tap **Earn more points** or dismiss.
- Modal should close.

## Scenario 18.2: Points multiplier

- Use campaign where `checkInCounter` is below multiplication limit.
- Complete check-in.
- Modal points should equal multiplier rate times 10.
- Use campaign where `checkInCounter` reached/exceeded multiplication limit.
- Complete check-in.
- Modal points should equal 10.

## Scenario 18.3: Post-success state

- Complete a successful check-in.
- Return to the same eligible item.
- Last check-in timestamp should affect button state.
- User should not be able to immediately receive points again.
- Relevant points/feed/check-in state should refresh after success.

## Scenario 18.4: Check-in API error

- Force or use a backend state where check-in API fails.
- Tap check-in from ready state.
- Congratulations modal should not open.
- Button should move to waiting/error-handled state.
- No points should be shown as awarded.
