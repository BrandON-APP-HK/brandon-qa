# 12. Search Deals

## Scope

Search tab access, query submit, sorting, pagination, refresh, detail, and deal actions.

## Preconditions

- Searchable social deals and promo blocks exist.
- Test as guest and logged-in user.

## Scenario 12.1: Guest Search tab guarded

- Start logged out.
- Tap **Search** tab.
- [ ] Login bottom sheet should open.
- [ ] Search screen should not open through the tab bar.

## Scenario 12.2: Submit search query

- Log in.
- Open **Search**.
- [ ] No results list should be displayed before submitting a query.
- Enter a search term.
- Tap search icon.
- [ ] Results should load for the submitted term.
- Clear or change the query and submit from keyboard.
- [ ] Results should update.
- Enter another query and blur the input.
- [ ] Blur should submit/update search when the query changed.
- Submit the same query again.
- [ ] Duplicate submit should not visibly reload unnecessarily.

## Scenario 12.3: Search sorting

- Submit a query.
- Open sort picker and choose **Rating**.
- [ ] Results should reload with Rating sort.
- Choose **Date**.
- [ ] Results should reload with Date sort.
- Choose **Distance** and grant location.
- [ ] Results should use current location for distance sort.
- Deny location and try Distance again.
- [ ] Sort should not change to Distance when location is denied.

## Scenario 12.4: Search refresh and pagination

- Submit a query with many results.
- Pull to refresh.
- [ ] Current query and sort should remain applied.
- [ ] Results should refetch.
- Scroll to the end.
- [ ] Next page should load when available.

## Scenario 12.5: Search result actions

- Submit a query.
- Tap a social deal image.
- [ ] `/search/social-deal-details` should open.
- Close details.
- Tap heart.
- [ ] Like/unlike should toggle.
- Tap report/flag.
- [ ] `/search/flag-deal` modal should open.
- Submit a reason.
- [ ] Modal should close and result should be flagged.
- Tap address/map action.
- [ ] `/search/map` should open with selected deal context.
